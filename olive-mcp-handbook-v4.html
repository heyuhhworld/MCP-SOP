0.1为什么需要 Claude Agent + MCP

私行 RM 每天面对的信息散布在多个系统中：基金详情在 IPMS，净值曲线在 Bloomberg，持仓在全委系统，询价在邮件里。一个看似简单的客户问题，背后往往需要跨越 4–5 个系统、花费 20–40 分钟才能给出像样的回答。这还不算整理过程中容易引入的数据出入与版本混乱。

Claude Agent + MCP 解决的正是这件事。Claude 作为 AI 大脑，理解用户的自然语言问题，自动规划应调用哪些数据接口；MCP 作为数据手臂，连接 Olive 底层各系统，实时取回结构化数据；Agent 再将这些数据组织成可视化、可对话的回答——整个过程对用户完全透明，像和一位随时在线的资深同事交谈。

0.3Olive MCP 是什么

Olive MCP 是 Olive 资产管理面向 AI Agent 的能力接口层，将产品知识、基金数据和全权委托组合信息以标准化工具形式开放给 AI。它不是一个独立产品，而是 Agent 智能体系中的"取数手臂"——Agent 负责理解用户意图、规划调用顺序、组织最终回答；MCP 负责连接底层数据系统，返回结构化的事实数据。

核心原则：工具只返回数据，推断由模型完成。MCP 不做评分、不做建议、不做判断——确保工具输出的客观性和可复用性。

Agent 框架的能力正以前所未有的速度迭代演进。

我们的判断是：不重复造轮子——将对话推理、工具规划、多步执行交给 Claude Desktop 等当下最强的 Agent 框架来做；而将有限的研发投入，聚焦于构建资管领域独有的数据资产与 MCP 调度能力：精准的产品数据模型、覆盖私募全生命周期的工具集、以及与合规体系深度绑定的权限架构。Agent 框架可以替换，但这套数据与调度能力，才是我们真正的护城河。

3.0协作总原则

Olive MCP 不是一个纯技术项目，也不是一个纯业务项目。它的成败，取决于业务团队和技术团队能不能配合好。在进入具体职责之前，先明确协作的总原则。

Business · 业务侧
Olive 业务团队
定义"做什么有价值"——基于一线工作流，识别真实场景，提出真实需求，验证真实效果。
×
Technology · 科技侧
Olive 科技团队
实现"怎么做才可靠"——把业务需求翻译成可执行的技术方案，保障安全、性能、可维护性。
Shared Accountability · 共同负责
"做出来好不好用"——这件事不是任何一方单独的责任。

这条原则有三个推论，它们决定了后面所有的协作规则：

  • 业务作为核心 Owner，对最终结果负责——业务方必须深度参与场景定义、工具描述、验收测试，直到最终上线。这不是"科技团队的项目"，是双方共建的项目，业务对结果负有不可转移的主责。
  • 技术方不能"做完交付就完事"——MCP 工具上线只是开始，持续根据业务反馈迭代描述、扩充评测集、优化召回率。Olive MCP 是产品，不是项目。
  • 双方必须有共同的判断标准——什么算"做得好"——工具召回率多少、用户满意度多少、误用率多少——必须有共同认可的可量化指标。没有共同标准，就没有共同负责。

3.1业务侧职责

MCP 不是科技团队的内部项目，业务团队是整个生命周期的核心驱动方。从场景定义、字段口径确认、数据权限审批，到验收签字和上线后的变更发起，业务团队全程参与，而非只在最后做验收。

  • ▸ 场景与需求识别——从一线工作流出发，识别哪些场景值得做 MCP 化、优先级如何排。
  • ▸ 工具语义定义——参与每个 MCP 工具的"业务定义"：这个工具回答什么问题、用什么真实话术、什么情况该调用、什么情况不该调用。这是业务专业的核心贡献。
  • ▸ 真实数据样本提供——为评测集贡献真实场景问答（脱敏），作为 MCP 工具质量的判断基准。
  • ▸ UAT 验收——业务侧负责人作为最终验收方，从"用得好不好用"角度判断 MCP 工具是否达标。
  • ▸ 上线推广与培训——业务侧主导使用培训、推广、最佳实践沉淀。
  • ▸ 持续反馈——日常使用中收集问题、整理高频痛点，作为下一轮迭代的输入。
  • ▸ 合规风险识别——合规代表对 MCP 工具的数据访问范围、输出形态进行合规审查。
一个观念升级：过去业务侧"提需求 → 等交付 → 验收"。MCP 时代，业务侧是每一个 MCP 工具的最终结果负责人，从场景定义到上线全程主导，不能止步于提出需求。这个角色升级，需要业务团队配置专门的 PM / BA 角色全程跟进。

3.2科技侧职责

Olive MCP 的科技侧分两个层次：Olive BU 技术团队（负责 Olive MCP 的具体实现）和中台架构团队（负责统一规范和治理底座）。两层分工的核心判断：业务定义的事，Olive 自己做；通用规则的事，中台统一做。

Olive BU Tech
Olive MCP 的设计与实现

  • ▸工具描述工程——按规范写好工具描述、参数说明、示例。这是技术活，但需要业务理解
  • ▸权限与脱敏实现——服务端硬编码行级权限、字段脱敏
  • ▸性能优化——查询响应时间、并发支持、错误处理
  • ▸评测自测——基于业务侧提供的样本，跑回归测试
  • ▸线上运维——监控、告警、问题排查、版本迭代

  • ▸审计与监控平台——全链路调用日志、异常检测、合规审计
  • ▸评测平台——评测集管理、自动回归、质量度量

  • ▸跨 BU 协调——工具命名空间、数据契约、与 CRM MCP 等的协同

3.4数据分级与权限体系

数据按敏感度分为四级（L1–L4），并通过三道递进的防线保障权限安全。

类型

原则：未分级数据一律不得接入 MCP。L3/L4 数据须经数据开放视图层过滤后才能进入工具层。

数据权限通过三道递进的防线来保障。每道防线有独立的触发时机与控制方式——任何一道失效都会影响整体安全性。

①
第一道 · 身份确认
你是谁？

在任何工具调用发生之前，系统首先确认当前用户的身份（client_id / rm_id / 角色），建立本次会话的权限基线。RM 调用客户相关工具前，还须通过授权关系校验。

②
第二道 · 数据范围限定
你能看什么？

数据开放视图层在返回信息前，依据用户角色和数据等级裁剪超出权限的字段。L3 数据返回前须确认「只看自己」；L4 数据不进入 MCP 工具链。

③
第三道 · 输出合规过滤
最终呈现什么？

最终回答在输出前经过合规话术检查，确保用户界面不出现违规内容；涉及预测性内容须附免责说明；估算数据须标注置信度。

核心原则：未完成分级标注的数据，一律不得接入 MCP 工具。三道防线中任何一道的变更，须经合规审批后方可上线。

3.5合规准入 Checklist

合规团队在审批阶段逐项确认，所有项目通过方可进入测试阶段：

数据安全
  • 所有出参字段已完成数据分级标注（L1–L4）
  • 不含 L4 数据（身份证号、完整银行卡号、税务 CRS 信息）
  • L3 数据已确认通过数据开放视图层过滤，并仅对授权角色可见
  • 工具不会返回超出请求者权限范围的数据字段
访问控制
  • 三道防线中的身份确认逻辑已明确（client_id / rm_id 传参规范）
  • RM 访问客户 L3 数据须有授权关系校验，逻辑已设计并审核
  • 工具调用日志已纳入审计链路，可溯源到调用者身份和时间戳
输出合规
  • 工具返回值为纯数据，不含评分、投资建议或预测性结论
  • 涉及估算或近似数据的字段已标注置信度或数据时效
  • 工具描述（Tool Description）中无误导性表述

3.6测试与上线检查

上线前须完成两个维度的测试，分别由业务团队和技术团队负责：

业务 UAT 场景验证
  • ·提供覆盖主要业务场景的真实问题集（不少于 10 个）
  • ·验证答案内容符合业务预期，信息准确无误
  • ·确认不同角色（客户/RM/投资经理）的数据可见范围正确
  • ·验证边界场景：无数据时的兜底回答是否合理
技术测试 功能与安全
  • ·接口功能测试：入参覆盖、出参格式、错误码验证
  • ·权限隔离验证：跨用户数据不可见性测试
  • ·边界与异常测试：空值、超范围参数、超时处理
  • ·多轮对话上下文传递测试（如涉及 existing_exposure 等上下文参数）

上线当天由技术负责人逐项确认，并在上线后 24 小时内完成稳定性观察：

上线前确认
  • 合规准入 Checklist 已全部通过，签字文档已归档
  • 业务 UAT 测试用例全部通过，无 Blocker 级问题未解决
  • 工具规格文档已更新至工具全览章节
  • 导言中工具总数统计已同步更新
  • 回滚方案已准备：可一键禁用该工具且不影响其他工具正常运行
上线后 24h 观察
  • 工具调用成功率 ≥ 95%，异常调用已分类记录
  • 权限拦截日志无异常（未出现越权访问尝试）
  • 一线用户（RM/投研）无 Blocker 级反馈
回滚触发条件：上线 24h 内出现以下任一情况，立即回滚并启动根因分析——调用成功率低于 80%、出现数据越权访问记录、业务团队发现数据错误影响客户服务。

4.1适用范围与触发条件

以下三类变更必须走本 SOP 流程，不得直接上线：

  • 新增工具。任何新的 MCP Tool 接入，无论数据等级高低，均须完整走完本 SOP。
  • 变更现有工具的入参或出参字段。包括新增字段、删除字段、字段含义/类型变更。若变更导致数据等级升高（如从 L2 升至 L3），须额外触发合规审批。
  • 数据来源或权限模型调整。底层数据源切换、访问控制逻辑变更、三道防线中任一防线的逻辑修改。
豁免情形：仅修改工具描述文案（不影响入参/出参/权限逻辑）可通过轻量审核——产品 + 技术负责人双签即可，无需完整 SOP 流程。

4.2工具提案与评审流程

从需求提出到合规审批，共五个阶段，每个阶段有明确责任方和交付物：

01
需求提案（RFC）
需求方填写工具提案文档，说明：业务场景和解决的问题、预期调用角色（客户/RM/投资经理）、需要读取哪类数据（初步判断数据等级）、与现有工具的区别（为什么不能复用/扩展参数）。
业务团队发起
02
产品评审
业务负责人确认场景价值与优先级，判断是否可以通过现有工具的参数扩展实现（避免工具增殖），明确目标用户角色和数据可见范围，输出"产品确认备忘"。
业务负责人
03
技术设计评审
技术团队完成工具规格设计：确定 Tool Name、入参/出参字段定义、数据来源与溯源映射、错误处理与边界行为说明。评审通过后输出"MCP 工具规格文档"，并更新至 Ch03 工具全览。
技术团队
04
合规审批
合规团队依据工具规格文档，完成 4.3 合规准入 Checklist 审核。L2 工具由合规团队审核签字；L3 工具须合规负责人签字；涉及 L4 数据的方案直接驳回，须重新设计。
合规团队管理层（L3+）
05
测试 → 上线
完成 4.4 测试准入要求后，走 4.5 上线检查清单，方可正式上线。上线后进入持续运营阶段（见 4.6）。

4.3合规准入 Checklist

合规团队在审批阶段逐项确认，所有项目通过方可进入测试阶段：

数据安全
  • 所有出参字段已完成数据分级标注（L1–L4）
  • 不含 L4 数据（身份证号、完整银行卡号、税务 CRS 信息）
  • L3 数据已确认通过数据开放视图层过滤，并仅对授权角色可见
  • 工具不会返回超出请求者权限范围的数据字段
访问控制
  • 三道防线中的身份确认逻辑已明确（client_id / rm_id 传参规范）
  • RM 访问客户 L3 数据须有授权关系校验，逻辑已设计并审核
  • 工具调用日志已纳入审计链路，可溯源到调用者身份和时间戳
输出合规
  • 工具返回值为纯数据，不含评分、投资建议或预测性结论
  • 涉及估算或近似数据的字段已标注置信度或数据时效
  • 工具描述（Tool Description）中无误导性表述

4.4测试准入要求

业务 UAT

4.5上线检查清单与回滚方案

上线当天由技术负责人逐项确认，并在上线后 24 小时内完成稳定性观察：

上线前确认
  • 合规准入 Checklist 已全部通过，签字文档已归档
  • 业务 UAT 测试用例全部通过，无 Blocker 级问题未解决
  • 工具规格文档已更新至 Ch03 工具全览
  • Ch00 工具总数统计数字已同步更新
  • 回滚方案已准备：可一键禁用该工具且不影响其他工具正常运行
上线后 24h 观察
  • 工具调用成功率 ≥ 95%，异常调用已分类记录
  • 权限拦截日志无异常（未出现越权访问尝试）
  • 一线用户（RM/投研）无 Blocker 级反馈
回滚触发条件：上线 24h 内出现以下任一情况，立即回滚并启动根因分析——调用成功率低于 80%、出现数据越权访问记录、业务团队发现数据错误影响客户服务。
